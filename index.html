// what is oklch?
// the problem with hsl

HSL is the colour model most designers learned first. It describes colour with three values: hue (the colour), saturation (how vivid), and lightness (how bright). But its lightness is a lie. A yellow at HSL 50% lightness looks far brighter than a blue at HSL 50% lightness. The numbers say they're equal — your eyes say otherwise.

This means that when you build a scale in HSL, visually pairing colours across that scale — say, using a step-400 blue next to a step-400 yellow — produces combinations that feel unbalanced even when the values match. You end up nudging by eye every time.

// what oklch fixes

OKLCH is a perceptually uniform colour space. Its L axis (lightness) is calibrated to how human vision actually works — so L 0.58 in a green and L 0.58 in a purple genuinely appear equally bright to the eye. Same number, same perception.

This has a practical consequence: when you set a rule like "all interactive elements use step 500", every colour in your palette at step 500 will feel visually consistent in weight, regardless of hue. No more manual nudging.

// the three axes
L — lightness

Runs from 0 (absolute black) to 1 (absolute white). Perceptually linear — each step feels like an equal jump in brightness. Step 50 in this tool sits at L ≈ 0.995, step 900 at L ≈ 0.10.

C — chroma

How vivid or saturated the colour is. 0 is a neutral grey. Higher values are more vivid. Unlike HSL saturation, chroma is absolute — C 0.18 means the same vividness regardless of hue or lightness.

H — hue

The angle on the colour wheel, 0°–360°. But unlike HSL, equal angular distances in OKLCH correspond to genuinely equal perceptual distances. Complementary colours are truly complementary.

// how this tool uses oklch

You pick a primary colour by setting H, C, and L — or by typing any hex code. The tool converts that hex to OKLCH to extract its true hue and chroma, then builds a 10-step scale by walking L from near-white (0.995) to near-black (0.10), keeping H constant and tapering C near the extremes so the lightest step approaches white and the darkest approaches black.

Secondary colours are derived by rotating H by a fixed angle (30°, 120°, or 180°) while keeping C and the same L stops — so they're harmonically related and visually equal in weight at every step. Status colours (danger, warning, success, info) use canonical hues that shift slightly if your primary is too close, so they always read as distinct.

The result is a palette where any two colours at the same step number are genuinely comparable in brightness — a property you simply cannot guarantee in HSL.

// browser support

The oklch() CSS function is supported in all modern browsers as of 2023 — Chrome 111+, Firefox 113+, Safari 15.4+. For older environments, the exported CSS includes hex fallback values on every line. The Tailwind and Figma exports use hex throughout.
